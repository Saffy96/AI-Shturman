# Отчёт по отметкам конкретной АЗС

Дата проверки: 13 июля 2026 года. Тестовая станция: `osmId=1077518659`.

## Текущий поток данных

Карточка АЗС лениво вызывает backend endpoint `GET /api/fuel/stations/:osmId/details` только при раскрытии. Backend параллельно вызывает:

- `GET https://gdebenz.ru/api/comments/{osmId}`;
- `GET https://gdebenz.ru/api/comments/{osmId}/recent?limit=30`.

До исправления использовался `Promise.all`, recent запрашивался с `limit=12`, а UI дополнительно показывал только первые 6 элементов без режима полного списка.

## Фактический контракт ГдеБЕНЗ

`/comments/{osmId}` возвращает объект-агрегат состояния, а не массив комментариев. В проверенном ответе присутствовали `status`, `confirmations`, `confirmationsFresh`, `realCount`, `updated`, `fuelsNow`, `pricesNow`, `limited`, `limits`, `addr` и другие поля. Значение `realCount` было равно 37.

`/comments/{osmId}/recent` возвращает массив событий напрямую. Поля события: `status`, `detail`, `created_at`, `edited`, `author_reliable`, иногда `on_site`. Метаданных `total`, `limit`, `offset` или `page` в ответе нет. Временные поля без суффикса зоны содержат UTC, как и предполагает frontend ГдеБЕНЗ.

Frontend ГдеБЕНЗ всегда добавляет `fp=<fingerprint>`. Без `fp` endpoint отвечает с `Cache-Control: public, max-age=1800` и может возвращать устаревшее на десятки минут окно. AI Штурман теперь передаёт стабильный read-only fingerprint, получает свежий вариант ответа и затем ограничивает нагрузку собственным 90-секундным кэшем.

Проверенные значения `limit`:

| Запрос | Получено |
| --- | ---: |
| `limit=12` | 12 |
| `limit=30` | 30 |
| `limit=50` | 30 |
| `limit=100` | 30 |

Параметры `offset=30`, `page=2` и `skip=30` были проигнорированы: сервер снова вернул те же 30 событий. Поэтому 30 — фактическое доступное окно истории, хотя агрегат сообщает о 37 отметках. Клиент не имитирует отсутствующую пагинацию и показывает все фактически полученные события.

Порядок массива `/recent` не является хронологическим: в проверенном ответе события «на месте» шли отдельным блоком перед частью более свежих событий не на месте. Сортировка ответа API использоваться не должна.

## Результат исправления

- Оба upstream endpoint вызываются через `Promise.allSettled`; данные исправного endpoint сохраняются при ошибке второго.
- Ответы-массивы и обёртки `data`, `items`, `comments`, `recent` поддерживаются единым извлечением записей.
- Все события приводятся к `StationActivity`; неизвестный тип сохраняется как `unknown`.
- Даты читаются из всех согласованных полей, Unix seconds и milliseconds различаются, невалидная дата не заменяется текущим временем.
- Дедупликация сначала использует source id, затем составной ключ. Одинаковый текст в разное время и разные типы в одно время не удаляются.
- После общего merge выполняется единая сортировка от новых событий к старым; неизвестные даты оказываются в конце.
- API-клиент больше не обрезает историю до 12, UI показывает превью из 5 и полный список из всех полученных записей.
- Полный список группируется по локальной дате устройства и показывает относительное время для свежих событий.
- Детали кэшируются на 90 секунд. Кнопка «Обновить» принудительно обходит кэш.
- Raw и нормализованные ответы логируются только в development. Счётчики этапов также доступны только в development UI.
- Диапазоны очереди вроде `50–100 машин` исключаются из разбора марок топлива и не создают ложный `АИ-100`.

## Контрольная сверка

Результат live-проверки нормализованного ответа:

| Поле | ГдеБЕНЗ | AI Штурман |
| --- | --- | --- |
| Всего отметок | `realCount=37`, history endpoint доступно 30 | 30 фактически полученных и отображаемых |
| Самая свежая | определяется свежим `/recent` с `fp` | совпадает после преобразования UTC в локальное время устройства |
| Самая старая в доступном окне | определяется текущим 30-элементным окном | совпадает с ГдеБЕНЗ |
| Очередь | сохраняется в `status/detail` | тип, текст, топливо и диапазон машин сохраняются |
| Лимит | сохраняется в `detail/limits` | тип/текст и число литров сохраняются |
| Был на месте | `on_site` | `wasOnSite`, отображается бейджем |

При контрольной проверке с `fp` свежая запись имела `created_at=2026-07-13 15:56:03`, то есть 18:56 по Москве. Автоматическая проверка подтвердила строго убывающий порядок timestamp.
